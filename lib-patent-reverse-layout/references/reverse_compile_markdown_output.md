# Markdown 最终申请稿合成规则

用于 `Reverse_Compile` 阶段。目标是把 Step 1-4 已确认内容合成为可直接审阅、可继续改写、可转入附件母稿改写流程的 Markdown 申请稿。

## 一、输出文件

默认文件名：

```text
[主题]_最终申请稿.md
```

若用户指定文件名，必须使用用户指定名称；若用户未指定路径，优先输出到当前项目目录。Compile 阶段默认只输出 Markdown 文件，不调用文档排版、页面渲染或文字处理软件流程。

## 二、标题结构

最终申请稿采用以下一级标题顺序：

```markdown
# [发明名称]

## 权利要求书

## 说明书

### 技术领域

### 背景技术

### 发明内容

#### 要解决的技术问题

#### 技术方案

#### 有益效果

### 附图说明

### 具体实施方式

#### 一、关键术语和参数定义

#### 二、实施例

#### 三、对比例

#### 四、性能测试

#### 五、性能数据

#### 六、性能解释

### 产品反向鉴定方法

### 覆盖性审核表

## 摘要
```

可根据案件类型增删三级标题，但不得打乱“权利要求书、说明书、摘要”的大顺序。

## 三、内容合成要求

1. 权利要求书采用 Step 1 已确认版本，不得在 Compile 阶段自行改窄核心范围。
2. 实施例和对比例采用 Step 2 已确认版本；若含预言性数据，保留 `[P]` 标记。
3. 性能测试采用 Step 3 已确认方法，测试条件、计算公式和判定阈值必须唯一。
4. 产品反向鉴定方法采用 Step 4 已确认版本；若 Step 4 被用户明确跳过，写明“本节可作为可选补充实施方式”。
5. 表格优先使用 Markdown 表格；长表允许拆分为“结构参数表”“性能数据表”“权要覆盖表”。最终申请稿表格只保留专利文本需要的技术列，不保留内部管理列，例如“数据标识”“规避类型”“证明目标”。若用户要求在Markdown阶段区分预测值，将`[P]`附着在具体数值后，而不是单独设置数据标识栏；若用户要求正式稿或清洁版，则按对应正式输出规则删除`[P]`。
6. 正文不得保留商业型号、内部项目代号或用户明确要求排除的旧稿名称；应改写为“实施例”“对比例”“对照样”等中性编号。

## 四、性能测试合成要点

当性能指标为高倍率平台电压差时，统一写作：

```text
平台电压差为同一SOC或同一容量区间下，低倍率参考平台电压与对应倍率平台电压之差。
```

若使用 20A 至 40A 平台电压差增量斜率，按下式写入：

```text
S20-40 = (Delta V_40A - Delta V_20A) / 20
```

温度数据可作为辅助效果说明；除非用户明确选择温升/散热校准为本案核心，不得把温度校准模型替代结构参数作为独权核心。

## 五、Compile 自检清单

输出前逐项检查：

| 检查项 | 要求 |
|---|---|
| 阶段确认 | Step 1-4 均已确认，或 Step 4 被明确跳过 |
| 标题结构 | 一级标题完整，权利要求书在说明书之前，摘要在末尾 |
| 权要一致 | 独权核心参数、范围和术语与 Step 1 确认版一致 |
| 数据标记 | 实测数据和 `[P]` 数据未混淆 |
| 表格列 | 实施例、对比例和性能数据表未保留“数据标识”“规避类型”“证明目标”等内部管理列 |
| 测试口径 | 每项性能只有一个测试定义和计算口径 |
| 反向可测性 | 核心结构参数在鉴定方法中有对应检测路径 |
| 脱敏 | 商业型号、客户名、用户要求排除的旧稿名称未进入正文 |
| 避让旧主线 | 未把用户要求避开的旧案公式、旧案评价模型写成核心发明点 |

## 六、最终回复

完成 Compile 后，最终回复只需给出 Markdown 文件路径、关键检查结果和未完成事项。若用户需要按附件母稿改写，提示可进入 `Reverse_AttachmentRewrite`。
