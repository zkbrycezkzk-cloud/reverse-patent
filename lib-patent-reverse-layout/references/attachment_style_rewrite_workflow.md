# 附件母稿格式改写工作流

用于把已确认的完整申请稿或 Compile 文本，改写到用户提供的附件母稿、模板或既有 `.docx` 的标题和表格栏目下，并保持附件的格式、栏目顺序和代理文风。

## 触发场景

使用本参考的典型请求包括：

1. “读取附件并理解内容和格式，将完整申请稿内容按附件标题下内容替换与补充”；
2. “格式与行文风格与附件保持一致”；
3. “不要使用某个旧稿，从当前完整申请稿重新按附件改写”；
4. “把性能解释改成附件中逐变量、带数据对比的写法”。

若用户明确排除某个文档，不得读取或引用该文档作为内容来源。

## 输入识别

先确认三类输入：

| 类型 | 作用 | 处理方式 |
|---|---|---|
| 附件母稿 | 提供标题、表格、栏目、行文风格 | 只作为格式和风格依据，除非用户明确要求继承内容 |
| 完整申请稿 | 提供已确认的权利要求、实施例、测试和鉴定内容 | 作为技术内容主来源 |
| 用户新指令 | 约束口径，例如 Ts 定义、Z1/Z2定义、竞品数据是否预言性 | 优先级最高 |

不得覆盖附件母稿和完整申请稿；输出应另存为新版文件。

## 标题映射规则

读取附件后，建立映射表：

| 附件栏目 | 填入内容 |
|---|---|
| 背景技术/现有技术不足 | 从完整申请稿背景和技术问题压缩改写 |
| 技术方案/详细方案 | 权利要求、关键参数定义、制备方法、实施例、对比例、测试方法和数据表 |
| 有益效果 | 用性能联立约束、结构联立条件、反向可测性和覆盖性概括 |
| 核心保护点/创新点 | 独权组合、关键从权、可反向检测路径、数据居中范围 |
| 替代方案/等同方案 | 材料替代、结构替代、测试路径替代、参数范围替代 |

若附件是表格型交底书，应尽量保留原表格结构，在对应空白单元格或正文单元格中替换补充内容。

## 性能解释写法

性能解释不得只写抽象机理，应学习附件中“由表1可知”的句式，按变量展开：

```text
由表1可知，实施例S1-Sn均能够同时满足[性能阈值]和[结构阈值]的综合约束；对比例C1-Cm则至少在[指标]之一上偏离上述范围。上述结果说明，本发明并非单纯追求[单一效果]，而是通过[材料/极片/结构参数]的协同设计，使[机理通道]处于匹配状态。

（一）[变量名称]的影响。对比实施例Sx、Sy与对比例Cz可知，当[变量]处于[范围]时，[性能1]由[数值]变化至[数值]，[性能2]由[数值]变化至[数值]，均仍处于本发明限定范围；当[变量]进一步偏离至[数值]时，对比例Cz的[性能]恶化至[数值]。其原因在于，[机理解释]。因此，[范围]能够兼顾[效果1]和[效果2]。
```

每个核心变量至少说明：

1. 选取哪些实施例和对比例进行对比；
2. 指标如何变化；
3. 为什么变化；
4. 该变量范围如何支撑权利要求。

## 数据和术语一致性

改写时必须复核：

1. 已确认的性能口径不得混用，例如确认 `Ts≤66.5℃` 后，不得在权1或判定口径中改回 `ΔT≤7.5℃`；
2. `Z1` 和 `Z2` 的位置定义必须出现在权利要求或关键参数定义中；
3. 预言性数据保留 `[P]` 标记，除非用户明确要求转化为正式申请文本表述；
4. 竞品名称、我方代号、产品型号等商业信息在正式申请稿中应改写为样品编号、实施例编号或对比例编号；
5. 表格数据、段落数据、权利要求阈值三者不得互相冲突。

## DOCX处理和校验

当目标是 `.docx`：

1. 优先使用 `python-docx` 或文档技能读取附件表格、段落和样式；
2. 保留附件版式，不直接重建无关格式；
3. 生成后检查表格数量、目标单元格字数、关键标题和关键阈值是否存在；
4. 按 Documents 技能尝试渲染为页面图片进行视觉校验；
5. 若本机缺少 `soffice/LibreOffice` 或 Word 渲染链，执行结构校验并在最终回复中说明未完成视觉渲染。

## 输出命名

默认命名：

```text
[原完整申请稿主题]_按附件格式改写版.docx
```

如用户给出指定名称，以用户指定名称为准。
